# 可用性测试完整指南

## 测试类型

### 形成性测试（Formative Testing）
- 在设计过程中进行
- 目的：发现设计问题，及时修正
- 样本量：3-5人
- 时机：低保真阶段即可开始

###  summative 测试（Summative Testing）
- 设计基本完成后进行
- 目的：量化评估设计方案
- 样本量：8-12人
- 时机：高保真或最终产品

## 测试流程

### 1. 测试准备

**招募用户**：
- 目标用户群5-8人
- 覆盖不同特征（年龄、经验、频次等）
- 签署知情同意书

**准备材料**：
- 测试脚本（含开场语、任务、结束语）
- 原型或产品
- 记录工具（录屏、笔记）
- 问卷（SUS/SEQ）

**场地布置**：
- 被测者与主持人同侧
- 录屏设备对焦屏幕
- 保持安静，减少干扰

### 2. 测试执行

**标准开场白**：
> "今天我们要测试的是[产品名称]，这不是你的问题，是产品的问题。如果你遇到困难，我会等你，但不会帮助你。"

**出声思考法（Think Aloud）**：
> "请在操作过程中说出你在想什么、看到了什么、想做什么。"

**任务设计原则**：
- 每个任务有明确目标
- 任务可独立完成
- 时长控制在5-10分钟/任务
- 总任务数不超过8个

### 3. 测试记录

记录内容：
- ✅ 任务完成率（Completed / Partial / Failed）
- ⏱ 任务完成时间
- ❌ 错误次数
- 😰 困惑点（用户犹豫、皱眉、提问的地方）
- 💡 观察到的有益行为

## SUS量表（System Usability Scale）

**评分方式**：每题1-5分，换算后累加×2.5

| 题号 | 正向题（1-5分） | 反向题（5-1分） |
|------|----------------|----------------|
| 1 | 我愿意使用这个系统 | |
| 2 | | 我觉得这个系统很复杂 |
| 3 | | 我觉得这个系统很容易 |
| 4 | 我认为需要技术支持下才能使用 | |
| 5 | 各个功能整合得很好 | |
| 6 | | 我觉得很多不一致 |
| 7 | 我想象大多数人会很快学会 | |
| 8 | | 我觉得很麻烦 |
| 9 | | 我很有信心 |
| 10 | | 我需要学很多 |

**解读**：
- 90-100：优秀（Top 1%）
- 85-90：优秀
- 70-85：良好
- 68：平均分
- 60-70：及格
- <60：不及格，需要改进

## 问题优先级

**严重度分级**：

| 等级 | 定义 | 响应要求 |
|------|------|---------|
| P0 | 导致任务失败，无法继续 | 立即修复 |
| P1 | 造成明显困难，需多次尝试 | 本轮迭代修复 |
| P2 | 小障碍，可绕过但影响体验 | 下轮迭代修复 |
| P3 | 轻微不便，可忽略 | 记录，不紧急处理 |

**优先级 = 发生频率 × 严重度 × 影响范围**
